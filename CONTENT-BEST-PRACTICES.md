## Writing Style & Voice

### Core Voice
Direct, analytical, occasionally wry. Think: smart friend cutting through the noise.
We have real data. Let it do the work. No manufactured drama.

**DO:**
- Use contractions (it's, don't, won't, they're)
- Vary sentence length dramatically. Short punches. Then longer explanations that give the reader room to breathe.
- State opinions confidently when backed by evidence
- Let strong data speak for itself — no commentary needed after a number
- Use "you" to speak directly to the reader's situation
- Use specific numbers: "393 speakers" not "hundreds of speakers"
- Admit limitations occasionally — it builds trust

**DON'T:**
- Start sentences with "Basically" or "Essentially"
- Add empty affirmations after data ("That's impressive." — cut it)
- Over-explain when one sentence works
- Use dramatic framing that adds nothing ("But here's what they buried...")
- Uniform section lengths — vary dramatically based on importance
- Transition bridges: "So," "Meanwhile," "Additionally" — start with the punch

### Voice Check
Read it aloud. If it sounds like a person talking to a colleague, you're on track.
If you hear yourself performing casualness, rewrite it.

## AI Writing Tells — NEVER Use These

### 1. Em-Dashes
DO NOT USE. Use periods, commas, or restructure.
- BAD: "The data is live — and it changes everything."
- GOOD: "The data is live. It changes everything."

### 2. The False Reframe (Pseudo-Pivot) — HARD RULE, ZERO TOLERANCE
Negating something simple, then reframing it as something grander.
- BAD: "KeynoteData isn't a database. It's intelligence."
- BAD: "This isn't just speaker data. It's competitive advantage."
- BAD: "Not X, it's Y." / "Less about X, more about Y." / "isn't X. It's Y."
- BAD: "Built for people who plan events, not just attend them."
- GOOD: Just say what it IS. Drop the negation entirely.

### 3. Unearned Declarations
Phrases that claim clarity the writing hasn't earned.
- BAD: "The pattern here is clear:", "Here's the thing:", "The bottom line is this:"
- BAD: "What this means is:", "The takeaway:", "X made one thing clear:", "One thing is certain:"
- BAD: "The numbers:" (just state the numbers)
- BAD: "That matters because" (just state what matters)
- BAD: "X matters." (if you're writing about it, obviously it matters)
- GOOD: Just state it. No preamble.

### 4. Either/Or + Hedge
Setting up a false binary, then collapsing it.
- BAD: "That's either flexibility or uncertainty. Probably both."
- GOOD: Delete it.

### 5. Empty Calorie Words
- BAD: genuinely, truly, really, actually, quite, extremely, "X, full stop.", "continues to"
- GOOD: Cut the modifier. "This is early." not "This is genuinely early."

### 6. Patronizing Your Audience
- BAD: "This is something to keep in mind." / "You'll want to think carefully about this."
- GOOD: Delete. State facts. Trust your audience.

### 7. Dramatic Dichotomies
- BAD: "That's exciting if you want that, terrifying if you don't."
- GOOD: Describe what it is without the drama.

### 8. Tautologies
- BAD: "Time will tell." / "Success depends on execution." / "It remains to be seen."
- GOOD: Delete. If you can't say something specific, cut it.

### 9. Wordiness / Saying Things Twice
- BAD: "It cuts costs by 10x. If you're running at scale, your bill just got a lot more manageable."
- GOOD: "It cuts costs by 10x, making it cheaper at scale."
- Rule: If the second sentence restates or explains the first, cut it or combine them.

### 10. AI Structural Tells
- Uniform section lengths (every section is 4-5 paragraphs) — vary based on importance
- Hyper-dense stat clustering (spec-sheet dump) — spread stats throughout
- Rhythmic scannable structure (every paragraph perfectly balanced) — vary pacing
- Transition words as bridges ("So...", "Meanwhile...", "Additionally...") — start with the punch
- "Bigger picture:" / "For context:" / "Looking ahead:" — jump straight to the point
- Short dramatic openers ("The timing wasn't subtle.") — make them specific

### Banned Words
robust, leverage, synergy, holistic, cutting-edge, "in today's market",
"navigate the landscape", "game-changer", "paradigm shift", "continues to"

### What Reads as Human
- Punchy metaphors: concrete comparisons that reframe something familiar
- Visceral imagery: "a literal money pit" instead of "unprofitable"
- Specific details only someone in the industry would know
- Varied paragraph lengths — some one sentence, some longer
- Specific to THIS situation, not generic to all situations

## Copywriting Fundamentals

### The Three Rules (Harry Dry / DataStackGuide)
1. **Can I visualize it?** Concrete, vivid images stick. Abstract doesn't.
2. **Can I falsify it?** "393 speakers tracked" is falsifiable. "Comprehensive data" is not.
3. **Can nobody else say this?** If a competitor could say the same, you haven't differentiated.

### The SO WHAT Chain
Feature → Functional benefit → Financial benefit → Emotional benefit. Write from the emotional end.
- "lots of data" → "887 speakers, 43% C-level, all with LinkedIn"
- "many conferences" → "13 conferences you can research tonight"

### Copywriting Rules
- Write with your eraser. A great sentence is a good sentence made shorter.
- No one cares what you can do. Everyone cares what you can do for them.
- Avoid passive voice. Use active, direct language.
- Avoid "landing page words": unlock, unleash, enhance, exceed, empower, supercharge.
- Find the tension. "Pleasant" gets forgotten. Conflict creates interest.
- Every line should lead to the next (slippery slide).
- More periods, fewer commas. Short sentences. We like short sentences.
- Kill adverbs. Kill adjectives. Use concrete nouns and active verbs.

### CTA Copy: Call-to-Value
- BAD: "Get Started" / "Sign Up" / "Get Access" / "Get Early Access"
- GOOD: "See All 393 INBOUND Speakers" / "Get the Full Sponsor List" / "Export to CSV"
- The CTA should state what they GET, not what they DO.
- Specific beats generic. "See All 393 Speakers" beats "See All Speakers".

## Landing Page Formula (Harry Dry)

### Above the Fold
1. **Title** — Explain the value. Use a hook (Value + Objection removed) when not unique.
   - Under 8 words / 44 characters
   - Caveman test: glance and grunt back what you offer
   - Hook formula: Value → Objection → Value + Hook combined
2. **Subtitle** — Get specific. How does it create the value? Introduce the product.
3. **Visual** — Show the product. Real data, not abstract illustrations.
4. **Social Proof** — Instant credibility. Specific numbers beat vague claims.
5. **CTA** — Call-to-value. First-person outperforms ("See My Data" > "See Data").

### Below the Fold
6. **Features & Objections** — Make the value concrete. Address why they think they can't get it.
7. **More Social Proof** — Let data/outcomes sell it. Specific numbers.
8. **FAQ** — Tie up loose ends. Reframe objections as questions.
9. **2nd CTA** — Remind them why they're clicking. Reinforce the value.
10. **Founder's Note** — Put yourself in their shoes. Explain their problem. Show the happy ending.

### Social Proof Principles
- Above the fold = credibility ("887 speakers tracked")
- Below the fold = inspiring action (outcomes, specific results)
- Specific over vague: "43% C-level" beats "senior executives"

## Content Structure for KeynoteData Pages

### Conference Detail Pages
Structure that answers ICP questions in order of importance:
1. **Header** — Name, location, stats (speakers/sponsors/sessions/attendees)
2. **Seniority breakdown** — "43% C-level or VP" — the specific, falsifiable claim nobody else has
3. **Top companies** — "HubSpot · Google · Salesforce · Gong" — scannable, specific
4. **Speaker table** — Top 10 visible, gated for full access
5. **Sponsor table** — If data exists
6. **Sessions** — If data exists
7. **FAQ section** — Who speaks, how to apply, what companies sponsor, when/where
8. **Related pages** — Links to roundup/role pages (via CONF_RELATED_PAGES)
9. **CTA** — "See All {N} Speakers" (call-to-value, specific count)

### Roundup Pages ("Best X Conferences 2026")
1. **Header** — Title + intro: why these conferences, what the ranking is based on
2. **Ranked list** — Numbered cards, each linking to conference detail page
3. **CTA** — Email gate for full database access

### Role/Audience Pages ("Conferences for VP of Marketing")
1. **Header** — Who this is for, what problem it solves
2. **Prose sections** — 2-3 sections covering what this role looks for in a conference
3. **Conference grid** — Relevant conferences with links to conference detail pages
4. **FAQ** — Role-specific questions (3 minimum)
5. **CTA** — Email gate

### Category Pages (existing pattern)
- Prose sections → sample speaker data → FAQs → "Conferences in database" grid → CTA
- FAQPage schema on every category page

### Case Study Structure
1. **Hero** — Curiosity-sparking subtitle (don't reveal the finding upfront)
2. **Meta bar** — Industry, Stage, Analysis Type, Records Analyzed
3. **Results upfront** — Show the payoff immediately (credibility before scroll)
4. **Challenge → Analysis → Key Finding → Supporting data → CTA**

## SEO Technical Requirements

### Meta Tags (Every Page)
- **Title:** 50-60 characters, keyword-first, format: `{Keyword} | KeynoteData`
- **Description:** 120-160 characters (optimal 150-158), action-oriented, unique per page
- **Canonical:** Always `https://keynotedata.com/{path}/` (never GitHub Pages URL)
- **OG tags:** og:type, og:url, og:title, og:description, og:image (1200×630px)
- **Twitter Card:** summary_large_image, title, description, image

### Meta Description Templates (Under 160 chars)

Conference detail page:
```
See all {N} {Conference} speakers, {N} sponsors, and session data. Filter by seniority, company, and function.
```

Roundup page:
```
{N} {category} conferences ranked by real speaker roster data. See who speaks, who sponsors, and who attends.
```

Role page:
```
Best B2B conferences for {role}. Real speaker and sponsor data across {N} events tracked by KeynoteData.
```

Category page:
```
{Topic} for B2B event marketers and PR teams. {N} speakers, {N} sponsors, {N} conferences tracked.
```

### Heading Hierarchy
- One H1 per page, unique
- H2s and H3s in sequential order — never skip levels
- Short paragraphs (2-4 sentences), one idea per paragraph

### Internal Linking
- 3+ internal links per page beyond nav
- Conference pages → roundup/role pages (CONF_RELATED_PAGES dict)
- Roundup/role pages → conference detail pages (in ranked list / conference grid)
- Category pages → all 13 conference detail pages ("Conferences in database" section)
- Use descriptive anchor text — not "click here" or "learn more"

## Schema Markup

### Conference Pages — Event Schema
```json
{
  "@context": "https://schema.org",
  "@type": "Event",
  "name": "{Conference Name}",
  "description": "{Description}",
  "location": {"@type": "Place", "name": "{City, Country}"},
  "organizer": {"@type": "Organization", "name": "{Conference Name}"},
  "url": "{conference website}"
}
```

### FAQPage Schema (Category + Conference Pages)
```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "Question?",
      "acceptedAnswer": {"@type": "Answer", "text": "Answer."}
    }
  ]
}
```
Rule: Schema questions MUST match visible FAQ content on the page exactly.

### BreadcrumbList Schema (All Inner Pages)
```json
{
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    {"@type": "ListItem", "position": 1, "name": "Home", "item": "https://keynotedata.com/"},
    {"@type": "ListItem", "position": 2, "name": "Conferences", "item": "https://keynotedata.com/conferences/"},
    {"@type": "ListItem", "position": 3, "name": "{Page Title}"}
  ]
}
```

### Organization + WebSite Schema (Homepage)
Already implemented in `html_head()` in build.py.

### Validate Schema
Test with Google's Rich Results Test: https://search.google.com/test/rich-results
